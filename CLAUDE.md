# Minerva

A full-stack TypeScript web app for managing an alchemical/crafting system — recipes, ingredients, inventory, scheduling, NPCs, spells, and skills. Likely a companion tool for a tabletop RPG or game.

## Tech Stack

| Layer | Technology | Version |
|---|---|---|
| Runtime | Node.js | 24.14.0 |
| Package manager | pnpm | 10.30.3 |
| Full-stack framework | Nuxt 4 | 4.3.1 |
| Frontend framework | Vue 3 | 3.5.29 |
| State management | Pinia + @pinia/nuxt | 0.11.3 |
| CSS | Tailwind CSS | 4.2.1 |
| UI primitives | Radix Vue | 1.9.17 |
| Icons | Lucide Vue Next | 0.575.0 |
| HTTP client | axios | 1.13.6 |
| Server runtime | H3 (Nuxt built-in) | 2.0.1-rc.14 |
| ORM | Drizzle ORM | 0.45.1 |
| DB driver | pg | 8.19.0 |
| Database | PostgreSQL | 17.4 |
| Language | TypeScript | 5.9.3 |
| Test runner | Vitest | 4.0.18 |
| HTTP testing | Supertest | 7.2.2 |

**Single package** at root — no pnpm workspaces.
**Module system:** ES modules throughout.
**Frontend source:** `app/` (Nuxt `srcDir`)
**API handlers:** `server/api/` (Nuxt H3 file-based routing)

### TypeScript Config
- Root `tsconfig.json` extends `.nuxt/tsconfig.json` (generated by Nuxt)
- Nuxt enables strict mode including `noUncheckedIndexedAccess: true`

## Database

```bash
pnpm db:push        # Push schema changes to DB (dev)
pnpm db:generate    # Generate Drizzle migration files
pnpm db:studio      # Open Drizzle Studio GUI
pnpm db:seed        # Seed the database
```

- **Schema:** `server/db/schema.ts`
- **Relations:** `server/db/relations.ts`
- **Migrations:** `server/db/` (drizzle-kit output)
- **Drizzle config:** `drizzle.config.ts`
- **Client init:** `server/utils/db.ts` (Drizzle with `pg` Pool)
- **DB URL selection:** `server/utils/databaseUrls.ts` — picks URL based on `NODE_ENV` (development / test / production)

### Schema — Enums

```typescript
pgEnum('PotionQuality',     ['NORMAL', 'HQ', 'LQ'])
pgEnum('IngredientQuality', ['NORMAL', 'HQ', 'LQ'])
pgEnum('TimeSlot',          ['MORNING', 'AFTERNOON', 'EVENING'])
pgEnum('TaskType',          ['GATHER_INGREDIENT', 'BREWING', 'SECURE_INGREDIENTS',
                             'RESEARCH_RECIPES', 'RESEARCH_SPELL', 'FREE_TIME'])
```

### Schema — Tables

**Recipe system:** `recipe`, `ingredient`, `recipeIngredient`, `potion`

**Inventory system:** `inventoryItem`, `potionInventoryItem`, `itemInventoryItem`, `item`, `currency`

**Scheduling system:** `weekSchedule`, `daySchedule`, `scheduledTask`, `taskDefinition`

**Character / world system:** `person`, `skill`, `spell`

All defined in `server/db/schema.ts`, relations in `server/db/relations.ts`.

### Pinia Stores
One store per domain in `app/stores/`: `ingredient`, `inventory`, `item`, `people`, `potion`, `recipe`, `scheduler`, `skills`, `spells`. Stores use `axios` directly.

### Composables
- `useSearch` — Search/filter logic (supports `customFilter` for array fields)
- `useConfirm` — Confirmation dialog integration
- `useToast` — Toast notification integration

All composables are auto-imported by Nuxt. Explicit imports use the individual file paths.

### Component Organization
- `layout/` — Structural shell (imported explicitly via barrel)
- `ui/` — Generic Radix Vue-based primitives (imported explicitly via barrel)
- `features/` — Domain-specific components (imported explicitly via barrel)

## Backend Patterns

- Each resource has its own directory under `server/api/` with one file per HTTP method
- File naming: `index.get.ts`, `[id].put.ts`, `[id]/favorite.patch.ts`, etc.
- Error handling: `handleUnknownError.ts` utility used in all route handlers
- ID parsing: `parseId.ts` utility for parsing string route params to integers
- Single Drizzle client instance exported from `server/utils/db.ts`
- H3 v2 notes:
  - `readBody<T>(event)` returns `Promise<T | undefined>` — always use `?? {}` fallback
  - `mockEvent(url)` in tests takes URL only (no method arg)
  - `noUncheckedIndexedAccess` is on — array access returns `T | undefined`
- Every Drizzle `insert` and `update` must set `updatedAt: new Date().toISOString()`
- See `server/api/CLAUDE.md` for route handler boilerplate and Drizzle patterns
- See `server/test/CLAUDE.md` for test infrastructure and how to register new routes

## Known Gotchas

- **Vue template inline `if`**: `@update:open="if (!$event) foo = null"` causes a parse error. Extract a named method instead.
- **Test router ordering**: In `server/test/helpers.ts`, register static sub-paths before dynamic `:id` routes (e.g. `/api/spells/:id/progress` before `/api/spells/:id`) or the dynamic segment swallows the static one.
- **New test routes**: Adding a new API handler does NOT automatically appear in tests. It must be manually imported and registered in `server/test/helpers.ts`.
- **Store refresh**: All store mutations call `get*()` after success to re-fetch from server. There are no optimistic updates.
- **`useSearch` and arrays**: The composable only searches `string | number` fields. For array-of-object fields, pass a `customFilter` option.
- **`db:push` column drops**: Drizzle will prompt interactively when a column is being dropped. Use `pnpm db:push --force` to bypass.
- **Component imports**: Components are not auto-imported in this project. Always import explicitly from the barrel (`@/components/ui/button`, `@/components/features/people`, etc.).
- See `app/CLAUDE.md` for frontend-specific patterns and gotchas.
